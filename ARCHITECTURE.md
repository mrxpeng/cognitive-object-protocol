# COP Architecture

## Data flow

```
Raw content / AI generation
          │
          ▼
    .cop.json file
    (blocks + relations + state + operations + views + sources)
          │
          ├──► copctl validate               → schema + reference + semantic validation
          │
          ├──► copctl render --to html       → human-readable review dashboard
          │        └── --view <viewId>       → filtered by block type (risk_view, executive_summary…)
          │
          ├──► copctl export --to markdown   → Markdown with human-readable state annotations
          │
          ├──► copctl context --target       → context packet JSON (minimal model input bundle)
          │        ├── --prompt              → ready-to-paste prompt (XML-wrapped content)
          │        ├── --prompt-out <file>   → write prompt to file (avoids stdout/stderr mixing)
          │        ├── --estimate-tokens     → rough token estimate (±25%; CJK may differ 2-4x)
          │        └── --include-instructions → opt-in to include instruction blocks
          │
          ├──► copctl apply-op <doc> <patch> → apply proposed operations via engine.ts
          │        └── --dry-run             → preview without writing
          │
          └──► copctl hash --target <id>     → SHA-256 hash for preconditions.target_hash
```

## Module structure

```
src/
├── types.ts      — TypeScript type definitions (CopDocument, CopBlock, …)
├── engine.ts     — Operation engine: applyOperations, hashEntity, applyJsonPatch, mergePatch
│                   (independent of CLI — can be used by MCP servers, tests, other tools)
├── validator.ts  — JSON Schema + reference + semantic validation
├── renderer.ts   — HTML and Markdown rendering
└── cli.ts        — IO layer only: reads files, calls engine/validator/renderer, writes output
```

## Object model

```
CopDocument
├── object          — metadata: id, type, title, language, owner, state
├── blocks[]        — semantic content units (claim, evidence, risk, task, decision, …)
├── relations[]     — typed edges between blocks (supports, contradicts, risk_of, task_for, …)
├── comments[]      — human / agent feedback on specific blocks or relations
├── operations[]    — append-only audit log (proposed → applied | rejected | reverted)
├── views[]         — rendering plans: which block types to show for which audience
└── sources[]       — evidence sources, URLs, papers, files
```

Note: Context packets are **derived/transient** objects generated by `copctl context`. They are NOT stored in the document.

## Trust model (v0.1)

```
untrusted
    │ (agent generates content)
    ▼
agent_generated
    │ (agent applies reasoning / cites evidence)
    ▼
reasoned
    │ (human reviews and approves — only human actors may advance)
    ▼
human_reviewed
    │
    └── (cryptographic verification: reserved for v0.2)
```

- `verified` trust_level is **removed from v0.1** (reserved for v0.2).
- Agents may NOT set trust_level to `human_reviewed`.
- The engine and validator enforce this rule.
- `actor.type = "human"` without `auth.verified` is workflow metadata only, not cryptographic proof.

## ID namespaces

| Prefix | Entity         |
|--------|----------------|
| obj_   | Object         |
| blk_   | Block          |
| rel_   | Relation       |
| cmt_   | Comment        |
| op_    | Operation      |
| view_  | View           |
| src_   | Source         |
| ctx_   | Context packet (transient, not stored in document) |

## Operation lifecycle

```
Agent reads block → cop hash --target → records preconditions.target_hash
          │
          ▼
Agent generates operations with status: "proposed"
          │
          ▼
Human or trusted system reviews
          │
    ┌─────┴─────┐
    ▼           ▼
"applied"   "rejected"
    │
    ▼
copctl apply-op → engine.ts → validates precondition hash → writes result
                           → re-validates whole document
                           → appends to operations[] audit log
```

## Concurrent edit safety

The CLI does not provide file-level locking. For concurrent multi-agent environments:

1. Record `preconditions.target_hash` before generating an operation.
2. Route `apply-op` through a serialised system (MCP server, database, git).
3. On hash mismatch, return conflict error — human resolution required.

## Protocol positioning

| System | Primary job | Relationship to COP |
|---|---|---|
| JSON-LD | Linked data interoperability | COP may export JSON-LD in v0.2 |
| Schema.org | Public structured-data vocabulary | COP may map block types to Schema.org types in v0.2 |
| Notion Block API | Workspace product blocks | Adapter target — import/export Notion pages as COP objects |
| MCP Resource | Runtime model context | COP objects can be exposed as MCP resources; engine.ts as MCP tools |
| RFC 6902 JSON Patch | Structured JSON edits | COP uses a safe subset (add/replace/remove) in operation patches |
